# Candidate Evaluation System — LLM-Driven Screening Agent (v5)

## 1. Objective

Design a system that evaluates whether a candidate possesses required competences while being robust to adversarial behaviors (e.g., LLM-assisted cheating).

The system estimates:

P(Acc = 1 | history)

Where:

- `Acc`: acceptance decision
- `history`: sequence of interview interactions

The system operates as a **sequential decision process with latent state estimation**.

---

## 2. System Overview

The system is a **POMDP with structured latent state**, implemented using LLMs for:

- Competence modeling
- Question generation (policy)
- Observation → evidence extraction

Core principle:

Belief_{t+1} = Belief_t + Evidence_t

---

## 3. Core Components

### 3.1 Role Description Format

All role descriptions must start with:

```
role_type: <student | professional>
name: <role_name>

[description content]
```

The system automatically detects `role_type` and loads appropriate prompts:

- **student**: Simplified, lenient prompts for educational assessment
- **professional**: Formal prompts for job interviews

---

### 3.2 Competence Model

A competence model is generated from a role description.

Structure:

```json
{
  "competences": [
    {"name": str, "weight": float}
  ]
}
```

Properties:

- Flat structure (no hierarchy)
- Number of competences: 3-5 for student mode, 5-8 for professional
- Weights normalized to sum to 1

---

### 3.3 Candidate Model (Ground Truth)

Defines the simulated candidate:

```json
{
  "competences": { "<name>": 0 | 1 },
  "behavior": "honest | cheater",
  "persona": "simple"
}
```

Used to generate a **candidate sheet** (observable profile).

---

### 3.4 Candidate Sheet

Generated by LLM and represents what the interviewer sees.

```json
{
  "summary": str,
  "experiences": [str],
  "claims": [str]
}
```

---

### 3.5 Belief State

Each competence is modeled as a Beta distribution:

- α_i: positive evidence
- β_i: negative evidence

Probability estimate:

p_i = α_i / (α_i + β_i)

Initial state:

α_i = 1, β_i = 1 (uninformative prior)

---

### 3.6 Evidence Model

Transforms (question, answer) into structured evidence.

#### Raw Evidence Mode (current implementation)

```json
{
  "competence": {
    "<name>": {"e_plus": float, "e_minus": float}
  },
  "behavior": {
    "cheating": float (professional) | "guessing": float (student)
  },
  "notes": str
}
```

Scoring differences:

- **student mode**: e_plus + e_minus ~ 0.2 (lenient, favor student)
- **professional mode**: e_plus + e_minus ~ 0.5 (strict but fair)

---

### 3.7 Belief Update

For each competence:

α_i ← α_i + e_i⁺
β_i ← β_i + e_i⁻

---

### 3.8 Action Model (Question Policy)

LLM selects next question based on:

- Competence model
- Candidate sheet
- Current belief state
- Conversation history

Output:

```json
{
  "question": str,
  "target_competences": [str],
  "type": "recall | application | practice" (student)
             "technical | behavioral | probing" (professional)
}
```

---

### 3.9 Decision Model

Decision is deterministic:

score = Σ w_i * p_i

accept = score > threshold

Configurable:

- threshold (default: 0.7)
- competence weights

---

### 3.10 Explanation Model (Separate)

A separate LLM generates explanation post-hoc using:

- conversation history
- belief trajectory
- final decision

## 5. Interaction Loop

### 5.1 Main Loop

At each step:

1. Generate question (LLM) - with spinner
2. Candidate answers (human or simulator)
3. Observation model extracts evidence (LLM) - with spinner
4. Update belief state (deterministic)
5. Check termination (LLM) - with spinner

### 5.2 In-Interview Commands

- `/quit` or `/q` - Immediately quit, save trajectory with `forced_end=True`
- `/end` or `/e` - End interview, generate decision and explanation
- `/resample` or `/r` - Resample current question

## 6. Trajectory System

### 6.1 Run ID

Format: `<config_name>_<timestamp>`
Example: `arithmetic_operations_20260326_123456`

### 6.2 Directory Structure

```
output_dir/
└── <run_id>/
    ├── competence_model.json
    ├── candidate_sheet.json
    └── trajectory/
        └── <run_id>_trajectory.json
```

### 6.3 Trajectory JSON Format

```json
{
  "id": "<run_id>_trajectory",
  "run_id": "<run_id>",
  "config": {...},
  "competence_model": {...},
  "candidate_sheet": {...},
  "turns": [
    {
      "turn": 0,
      "action": {"question": "...", "target_competences": [...], "type": "..."},
      "answer": "...",
      "evidence": {...},
      "belief_after": {...}
    }
  ],
  "final_decision": {...},
  "explanation": {...},
  "forced_end": false
}
```

### 6.4 Forking and Continuing

- **continue**: Creates `<run_id>_cont_<timestamp>`, continues from last turn
- **fork**: Creates `<run_id>_fork_<turn_idx>_<timestamp>`, forks from specific turn

## 14. Execution Flow

### 14.1 Start Command

1. Load and validate role description
2. Detect prompt type (student/professional)
3. Load appropriate prompts
4. Generate run_id
5. Generate competence model (with spinner)
6. Generate candidate sheet (with spinner)
7. Initialize belief state
8. Loop through questions
9. Compute decision
10. Generate explanation (with spinner)
11. Save trajectory and artifacts

### 14.2 Continue Command

1. Load existing trajectory
2. Create new run_id with `_cont_` suffix
3. Continue from last turn
4. Same loop as start

### 14.3 Fork Command

1. Load existing trajectory
2. Create new run_id with `_fork_<turn_idx>_` suffix
3. Copy turns up to fork point
4. Reconstruct belief state
5. Continue from fork point

## 16. Summary

The system is:

- A **belief-driven interviewer**
- Powered by LLMs for reasoning
- Grounded by **Bayesian evidence accumulation**
- Designed for **adversarial robustness and interpretability**
- Supports **multiple prompt modes** (student/professional)
- Includes **trajectory persistence** for forking and continuing
- Features **interactive CLI** with spinners and rich output

---

## 17. Key Differences from v4

1. **Prompt System**: Two modes (student/professional) with automatic detection
2. **Role Description Format**: Required `role_type:` and `name:` headers
3. **Trajectory System**: Full persistence with run_id, forking, continuing
4. **CLI Commands**: start, continue, fork, list-runs
5. **In-Interview Commands**: /quit, /end, /resample
6. **Spinners**: yaspin for all LLM calls
7. **Forced End**: support for quitting without decision
8. **Output Directory**: Structured artifact storage
9. **Lenient Scoring**: Student mode uses e_plus+e_minus ~0.2
